5.4.2 Real Time DataBase

¿Qué es Realtime Database? 🤔​

Firebase Realtime Database es una base de datos NoSQL alojada en la nube. Almacena datos en un formato JSON y mantiene la sincronización en tiempo real con todos los clientes que estén conectados, de modo que cada cambio en la base de datos se refleja inmediatamente en todas las aplicaciones que estén leyendo esos datos.

Características clave 🔑

  • Sincronización en Tiempo Real ⏱️
    Cualquier cambio en la base de datos se propaga instantáneamente a todos los clientes conectados.

  • Estructura en Árbol (JSON) 🌳
    Los datos se representan como un gran objeto JSON anidado.

  • Seguridad basada en Reglas 🔒
    Puedes establecer reglas de lectura/escritura basadas en la autenticación de los usuarios u otras condiciones.

  • Integración con Firebase Authentication 🏷️
    Es muy sencillo restringir el acceso únicamente a los usuarios registrados.

  • Soporte Offline 📴
    Ofrece persistencia local opcional. Si un dispositivo se desconecta, las lecturas y escrituras se guardan y se sincronizan cuando se recupera la conexión.

  • Escalabilidad 📈
    Realtime Database es adecuada para prototipos o aplicaciones de tamaño pequeño y mediano con funcionalidades en tiempo real. Sin embargo, con las configuraciones y el diseño de datos correctos, puede llegar a cubrir necesidades de aplicaciones más grandes.

Servicios y funcionalidades ⚙️​

  • Lectura y Escritura en Tiempo Real 📡
    Permite suscribirse a cambios en un nodo de la base de datos y recibir datos de forma automática cada vez que algo se actualice.
    Uso de listeners como ValueEventListener y ChildEventListener para detectar cambios en nodos específicos.

  • Sincronización Multiplataforma 🔄
    Una misma base de datos puede alimentar apps en Android, iOS, Web y otros entornos (por ejemplo, un backend en Node.js).

  • Control de Seguridad con Reglas 🔐
    Reglas simples o avanzadas para definir quién puede leer/escribir.

    Ejemplo de regla básica que permite acceso solo a usuarios autenticados:

    {
      "rules": {
        ".read": "auth != null",
        ".write": "auth != null"
      }
    }

  • Soporte para Operaciones Atómicas y Transacciones 🔁
    Posibilidad de usar transacciones para actualizar datos de manera segura cuando existe riesgo de colisión de múltiples escrituras en un mismo nodo.

  • Soporte Offline 📴
    Se puede habilitar la persistencia en el cliente. Una vez hecho esto, la aplicación seguirá funcionando incluso sin conexión, y se sincronizará cuando la conexión se restablezca.

  • Escalado Automático 🚀
    Aunque tiene algunas limitaciones frente a Firestore en consultas más complejas, Realtime Database puede escalar para muchas aplicaciones basadas en datos en tiempo real.

¿Por qué usar Realtime Database en lugar de Firestore? 🔍​

Cloud Firestore es otra base de datos de Firebase que ofrece consultas más sofisticadas, escalabilidad a gran escala y una estructura de documentos y colecciones. Sin embargo, hay escenarios en los que Realtime Database puede ser más ventajosa:

  1. Actualizaciones en tiempo real de alta frecuencia 📢 Si tu aplicación requiere notificaciones de cambios muy rápidas (por ejemplo, chats, juegos en tiempo real, apps de presencia en vivo), Realtime Database es muy buena opción.
    Firestore también soporta tiempo real, pero Realtime Database fue diseñada específicamente para este propósito desde el inicio y puede manejar grandes volúmenes de pequeñas actualizaciones concurrentes con facilidad.

  2. Simplicidad en la Estructura de Datos 🛠️ En Realtime Database, tenemos un árbol JSON. Para ciertos desarrollos (como pequeñas apps de mensajería, tablones de anuncios o sistemas de estados), puede resultar más sencillo y directo.
    Firestore requiere pensar la información en términos de colecciones y documentos (similar a la estructura de colecciones en NoSQL), lo que puede añadir complejidad innecesaria si necesitamos únicamente un árbol muy directo.

  3. Coste 💰 Dependiendo del uso, en algunos escenarios Realtime Database puede resultar más económico, sobre todo si hacemos un gran volumen de lecturas breves.
    Firestore en ocasiones puede resultar más costoso si las consultas son muy frecuentes (aunque depende mucho de la arquitectura).
    No obstante, es importante analizar el plan de facturación de ambos servicios para tu caso específico.

  4. Menor Necesidad de Consultas Complejas 🧐 Si tu caso de uso no requiere filtros múltiples, índices compuestos o consultas muy elaboradas, Realtime Database puede ser más que suficiente.
    Firestore ofrece queries más potentes (where, array-contains, etc.) y mejor indexación. Pero si no necesitas tanto, Realtime Database resulta más sencilla de usar.
