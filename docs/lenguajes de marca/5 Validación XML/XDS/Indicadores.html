Indicadores

En XSD (XML Schema Definition), los indicadores permiten controlar cómo se organizan o repiten los subelementos en un elemento complejo, así como la posibilidad de agrupar definiciones de elementos o atributos para reutilizarlas. En XSD se emplean los siguientes indicadores...

  • Indicadores de orden (<xs:sequence>, <xs:choice>, <xs:all>) definen cómo aparecen los subelementos (en un orden estricto, uno de varios, o todos en cualquier orden).
  • Indicadores de ocurrencia (minOccurs, maxOccurs) controlan cuántas veces puede aparecer un subelemento (de 0 a unbounded).
  • Indicadores de grupo (<xs:group> y <xs:attributeGroup>) permiten agrupar definiciones repetitivas de elementos o atributos y reutilizarlas en diferentes tipos o elementos, reduciendo la duplicación y aumentando la consistencia del XSD.

Estos indicadores son herramientas clave para describir estructuras complejas en tus esquemas XSD, pues te permiten detallar la organización, repetición y agregación de los componentes del documento XML de forma controlada y escalable.

1. Indicadores de Orden​

Los indicadores de orden en XSD determinan cómo deben aparecer los subelementos dentro de un elemento complejo. Existen tres principales:

<xs:sequence>​

  • Impone un orden estricto: los subelementos deben aparecer en el orden que se declara.

  • Ejemplo:

    <xs:element name="suscripcion">
      <xs:complexType>
        <xs:sequence>
          <xs:element name="usuario" type="xs:string"/>
          <xs:element name="fechaInicio" type="xs:date"/>
          <xs:element name="plan" type="xs:string"/>
        </xs:sequence>
      </xs:complexType>
    </xs:element>

    Aquí, <usuario> debe aparecer primero, luego <fechaInicio>, y finalmente <plan>. Un XML como:

    <suscripcion>
      <usuario>ProGamerX</usuario>
      <fechaInicio>2024-01-01</fechaInicio>
      <plan>Premium</plan>
    </suscripcion>

    cumple con ese orden. Cualquier permutación diferente (poner <plan> antes de <fechaInicio>) rompería la validación.

<xs:choice>​

  • Permite que solo uno de los subelementos (o varios si se cambia la cardinalidad) aparezca en cada posición.

  • Ejemplo:

    <xs:element name="mensaje">
      <xs:complexType>
        <xs:choice maxOccurs="unbounded">
          <xs:element name="sms" type="xs:string"/>
          <xs:element name="email" type="xs:string"/>
        </xs:choice>
      </xs:complexType>
    </xs:element>

    Aquí, el elemento <mensaje> puede contener uno o más (por maxOccurs="unbounded") subelementos, pero cada uno será un sms o un email. Podrías tener un <mensaje> con:

    <mensaje>
      <sms>¡Hola!</sms>
      <email>Te contacté antes...</email>
      <sms>Te recuerdo nuestra cita</sms>
    </mensaje>

    Cada vez escoges uno de los subelementos posibles.

<xs:all>​

  • Permite que los subelementos aparezcan en cualquier orden, pero cada uno puede tener minOccurs de 0 o 1, y maxOccurs solo puede ser 1.

  • Ejemplo:

    <xs:element name="perfil">
      <xs:complexType>
        <xs:all>
          <xs:element name="nickname" type="xs:string" minOccurs="1"/>
          <xs:element name="email" type="xs:string" minOccurs="0"/>
          <xs:element name="pais"  type="xs:string" minOccurs="1"/>
        </xs:all>
      </xs:complexType>
    </xs:element>

    En este caso, <nickname>, <email>, y <pais> pueden aparecer en cualquier orden, pero cada uno puede aparecer 0 o 1 vez (o 1 para nickname y pais, y 0-1 para email). No puedes repetir <nickname> varias veces.

2. Indicadores de Ocurrencia​

Los indicadores de ocurrencia controlan cuántas veces puede aparecer un elemento o un grupo de elementos. Se usan comúnmente en los atributos minOccurs y maxOccurs dentro de <xs:element>, <xs:sequence>, <xs:choice>, <xs:all> y otros contextos.

minOccurs​

  • Especifica el número mínimo de apariciones de un elemento.
  • Valor por defecto: 1 (significa que es obligatorio).
  • Si pones minOccurs="0", el elemento es opcional.
  • Ejemplo: 
    <xs:element name="amigo" type="xs:string" minOccurs="0" maxOccurs="5"/>
    <amigo> podría no aparecer en el XML o aparecer hasta 5 veces.

maxOccurs​

  • Especifica el número máximo de apariciones de un elemento.
  • Valor por defecto: 1.
  • Acepta también la palabra unbounded para permitir un número ilimitado de ocurrencias.
  • Ejemplo: 
    <xs:element name="video" type="xs:string" maxOccurs="unbounded"/>
    Significa que el elemento <video> puede repetirse tantas veces como se desee (0, 1, 2, …, n).

3. Indicadores de Grupo​

Los indicadores de grupo permiten agrupar definiciones de elementos o atributos, de modo que podamos reutilizar esos grupos en varios lugares del esquema. Esto ayuda a mantener el XSD más organizado y evitar duplicaciones.

<xs:group name="...">​

  • Sirve para agrupar un conjunto de elementos (definidos con <xs:sequence>, <xs:choice> o <xs:all>).
  • Luego, se puede usar <xs:group ref="nombreDelGrupo"> para insertar ese grupo en varios sitios.

Ejemplo:

<xs:group name="grupoPersona">
  <xs:sequence>
    <xs:element name="nombre" type="xs:string"/>
    <xs:element name="apellido" type="xs:string"/>
  </xs:sequence>
</xs:group>

<xs:element name="empleado">
  <xs:complexType>
    <xs:sequence>
      <xs:group ref="grupoPersona"/>
      <xs:element name="sueldo" type="xs:decimal"/>
    </xs:sequence>
  </xs:complexType>
</xs:element>

<xs:element name="cliente">
  <xs:complexType>
    <xs:sequence>
      <xs:group ref="grupoPersona"/>
      <xs:element name="empresa" type="xs:string"/>
    </xs:sequence>
  </xs:complexType>
</xs:element>

En este ejemplo, definimos un grupo de elementos llamado grupoPersona (nombre, apellido). Lo reutilizamos tanto en <empleado> como en <cliente>, evitando duplicar esos mismos <xs:element> en ambos sitios.

<xs:attributeGroup name="...">​

  • Similar a <xs:group>, pero para atributos.
  • Permite agrupar conjuntos de atributos y luego referenciarlos en distintos complexType o <xs:element>.

Ejemplo:

<xs:attributeGroup name="baseAtributos">
  <xs:attribute name="id" type="xs:ID" use="required"/>
  <xs:attribute name="version" type="xs:string" use="optional"/>
</xs:attributeGroup>

<xs:complexType name="documentoType">
  <xs:sequence>
    <xs:element name="titulo" type="xs:string"/>
  </xs:sequence>
  <xs:attributeGroup ref="baseAtributos"/>
</xs:complexType>

<xs:complexType name="informeType">
  <xs:sequence>
    <xs:element name="autor" type="xs:string"/>
  </xs:sequence>
  <xs:attributeGroup ref="baseAtributos"/>
</xs:complexType>

Ahora, tanto documentoType como informeType comparten los atributos id y version definidos en el attribute group baseAtributos, sin tener que copiarlos dos veces en cada tipo complejo.
